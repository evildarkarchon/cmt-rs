<interfaces>
Existing Rust boundary from Phase 1: `src/main.rs` declares `pub mod domain;`, and `src/domain/mod.rs` currently hosts only a no-op domain marker. This plan should replace or extend that placeholder with a `pub mod settings;` export while preserving any existing no-op tests that still serve Phase 1.

Required domain identifiers for downstream plans:
- `AppSettings`: top-level settings aggregate.
- `LogLevel`: persisted values `DEBUG`, `INFO`, `ERROR`; display labels are handled by the UI plan. Unsupported loaded values, including `WARNING`, repair to the default `INFO` per 02-SPEC.md and resolved 02-RESEARCH.md guidance.
- `UpdateSource`: persisted values `both`, `github`, `nexus`, `none`; default may be supplied by the settings store from `download-source.txt` in Plan 02.
- `ScannerSettings`: booleans for Overview Issues, Errors, Wrong Format, Loose Previs, Junk Files, Problem Overrides, and Race Subgraphs, all defaulting to `true`.
- `DowngraderSettings`: booleans for `downgrader_keep_backups` and `downgrader_delete_deltas`.
</interfaces>
</context>

<decision_coverage>
- D-09: Domain repair must reset malformed JSON to defaults, save defaults, log the parse failure, and continue.
- D-10: Domain repair must silently repair syntactically valid partial settings from the UI perspective and expose only logs/test diagnostics.
- D-11: Domain repair must not salvage values from malformed JSON; malformed JSON is defaults-only reset.
- D-12: Domain repair must ignore unknown valid-JSON keys in memory and remove them on resave.
- D-13: Domain tests must parse JSON and assert exact keys/values, not whitespace, indentation, or object ordering.
- D-14: Domain tests must explicitly assert unknown keys are removed after save.
</decision_coverage>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| local settings file → domain parser | User-editable or corrupted JSON crosses into typed Rust settings. |
| domain model → serialized JSON | Repaired values are written back and may remove unknown/tampered keys. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-01 | Tampering | `src/domain/settings.rs` JSON parser | mitigate | Validate known keys individually, default invalid values, reject malformed/non-object roots, and test `settings_repair`. |
| T-02-02 | Information Disclosure | `settings_repair` diagnostics | accept | Diagnostics must mention setting keys/failure classes only; no secrets are in Phase 2 settings values. |
| T-02-03 | Repudiation | silent repair path | mitigate | Return test-visible repair diagnostics and rely on Plan 02 logging for malformed/repair events without showing UI dialogs. |
| T-02-04 | Denial of Service | malformed settings content | mitigate | Never panic on malformed JSON; return a reset/default path and avoid production `unwrap()`/`expect()`. |
</threat_model>
