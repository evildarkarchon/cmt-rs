# Phase 03 — UI Design Contract

> Visual and interaction contract for frontend phases. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

---

## Design System

| Property | Value |
|----------|-------|
| Tool | none — native Rust/Slint desktop UI; shadcn is not applicable because this is not a React/Next.js/Vite project |
| Preset | not applicable |
| Component library | Slint `std-widgets.slint` only; use `TabWidget`, `GroupBox`, standard `Button`/`CheckBox`/text controls where needed |
| Icon library | none in Phase 3; do not introduce icons unless a later reference tab requires them |
| Font | Slint platform default UI font; preserve conservative desktop appearance rather than introducing a custom font |

Design-system source: existing `ui/*.slint`, `AGENTS.md`, `03-CONTEXT.md`, and `03-RESEARCH.md`. Phase 3 is a non-visual foundation slice, so it must not introduce a new design language, manual picker UI, or final Overview/Scanner/F4SE panels.

## Spacing Scale

Declared values (must be multiples of 4):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Fine inline gaps inside status rows and compact text/action pairs |
| sm | 8px | Radio/control internals, compact row spacing, `GroupBox` internal padding |
| md | 16px | Default element spacing and gaps between related status fields |
| lg | 24px | Tab content padding; existing Slint placeholder tab padding |
| xl | 32px | Separation between major discovery/background sections when later tabs render them |
| 2xl | 48px | Major section breaks in larger dialogs such as future Downgrade/Archive workflows |
| 3xl | 64px | Page-level spacing only; avoid in Phase 3 integration surfaces unless needed for empty-state centering |

Exceptions: standard interactive row height may remain 24px where already established by `SettingsRadioOption`; minimum mouse/touch target for any new clickable status action is 32px high. Do not add a 44px touch-only exception for this desktop-first Slint port.

## Color

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `#202020` | Background and dominant tab surfaces; already used by Settings tab |
| Secondary (30%) | `#2a2a2a` | Group boxes, status panels, progress rows, and future cards/sidebar-like areas |
| Accent (10%) | `#4da3ff` | Selected radio indicator, focused control outline, active progress indicator, safe primary actions only |
| Destructive | `#c94141` | Destructive or file-changing actions only; Phase 3 has no destructive actions |

Accent reserved for: selected settings indicators, focused/active status affordances, active progress indicators, safe primary actions such as a future `Refresh Overview`/`Scan Game` trigger. Accent must not be applied to every clickable element. Discovery failures use neutral/error text styling rather than the accent color.

## Phase 3 Interaction Contract

### Discovery Status Contract

| State | User-visible status | Required behavior |
|-------|---------------------|-------------------|
| Not started | `Discovery not run` | Later tabs may display this as a neutral empty state before startup/refresh discovery completes. |
| Running | `Discovering Fallout 4 installation...` | UI remains responsive; no Slint callback may block on filesystem, registry, process enumeration, or INI parsing. |
| Partial success | `Discovery completed with missing optional paths.` | Valid game path may still have missing `Data` or `Data/F4SE/Plugins`; represent missing fields without treating discovery as failed. |
| Success | `Discovery complete` | Later tabs may render typed game path, Data path, F4SE path, INI paths, archive/module sets, manager name, manager executable path, and manager version. |
| Not found | `A Fallout 4 installation could not be found.` | Use the reference-compatible message. Do not show a Phase 3 file picker; later UI phases own prompts/remediation. |
| Unsupported platform | `This discovery operation is not supported on this platform.` | Non-Windows real operations return explicit typed unsupported results without panics. |

### Background Task Event Contract

| Event/status | User-visible contract | Interaction rules |
|--------------|----------------------|-------------------|
| Started | `{Task name} started` | Mark the related task busy; disable only the same task's duplicate start action, not the full app. |
| Progress with text | Show provided text exactly when safe and user-facing | Optional text may be present without counts; do not invent percentage, rate, or ETA in Phase 3. |
| Progress with counts | `{text} ({current}/{total})` | Show counts only when both values are available and meaningful. |
| Cancellation requested | `Cancelling...` | Preserve a distinct pending-cancel state; do not immediately show final cancellation. |
| Cancellation acknowledged | `Cancellation acknowledged` | Keep task busy until final `Cancelled` or `Completed` event arrives. |
| Completed | `{Task name} complete` | Clear busy state and allow follow-up actions. |
| Cancelled | `{Task name} cancelled` | Clear busy state without marking the operation as an error. |
| Failed | Safe typed error message | Clear busy state, keep partial safe state visible, and expose raw details only through diagnostics/logging. |

Task names for UI copy: `Discovery`, `Scan`, `Patch`, `Download`, `External Process`, and `Background Task`. These names map to typed worker task kinds defined in Phase 3, even when the actual Scanner/Patcher/Downloader workflows are implemented later.

### Process/Desktop Action Result Contract

| Operation | Success message | Failure message pattern |
|-----------|-----------------|-------------------------|
| Open URL | `Opened link` | `Could not open link. Copy the link and open it manually.` |
| Open path | `Opened path` | `Could not open path. Check that the path exists and try again.` |
| Launch external tool | `Launched tool` | `Could not launch tool. Check that the tool is installed and try again.` |
| Read version metadata | no success toast required | `Could not read version information.` only if a later visible workflow needs to report it; otherwise log diagnostics and use `0.0.0` fallback where specified. |

Adapters must return typed action result events. They must not show OS dialogs directly, silently fail, or report only to logs.

### Slint Integration Rules

- Worker threads send owned Rust domain events only; Slint handles, Slint models, and UI objects must not cross worker boundaries.
- UI updates caused by background work must enter the Slint event loop through the Phase 3 handoff seam using `slint::invoke_from_event_loop` or `Weak::upgrade_in_event_loop`.
- Slint markup may display simple status/progress fields later, but filesystem/process/discovery logic remains in Rust controllers/services/adapters.
- Any visible interim UI added in Phase 3 must preserve the existing `TabWidget` order and not replace placeholder Overview, F4SE, Scanner, Tools, Settings, or About workflows owned by later phases.
